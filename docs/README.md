# Документация проекта — Навигация

> Этот файл объясняет, как устроена база знаний проекта и где что искать.

## Как устроена система хранения знаний

У нас 3 уровня документации:

### Уровень 1: Автозагрузка (каждая сессия)
| Файл | Что внутри | Когда читается |
|------|-----------|----------------|
| `CLAUDE.md` (корень проекта) | Обзор проекта, workflow, структура, правила | Автоматически при старте каждой сессии |
| `MEMORY.md` (автопамять Claude) | Ключевые факты, MVP решения, правила сервера | Автоматически при старте каждой сессии |

### Уровень 2: Подробная документация (docs/)
| Файл | Что внутри | Когда читать |
|------|-----------|-------------|
| `project-overview.md` | Что за проект, для кого, какую проблему решает, функции, чего НЕ будет | Знакомство с проектом, онбординг |
| `business-requirements.md` | Бизнес-требования MVP и будущее, user flow, формулы расчёта | Перед разработкой новой функции |
| `pipeline.md` | Детальное описание всех 6 этапов обработки данных, SQL запросы, JSON форматы | При работе над логикой обработки |
| `database-architecture.md` | Все 9 таблиц, поля, связи, MVP упрощения | При работе с БД |
| `pdf-formats.md` | Анализ всех 9 PDF файлов из разных лабораторий, вариации форматов | При работе над парсингом PDF |
| `server-infrastructure.md` | Сервер, Docker контейнеры, CI/CD, правила доступа | При деплое или настройке инфраструктуры |

### Уровень 3: Живые документы (пополняются)
| Файл | Что внутри | Когда пополнять |
|------|-----------|----------------|
| `changelog.md` | История изменений: что, когда, почему | После каждого значимого изменения в проекте |
| `errors-and-solutions.md` | Ошибки и как их решили | Каждый раз когда столкнулись с проблемой и решили её |

## Другие важные файлы
| Файл | Назначение |
|------|-----------|
| `database/schema.sql` | DDL схема БД (source of truth для структуры таблиц) |

## Правила ведения документации

### Когда ОБНОВЛЯТЬ документацию:
- Перед добавлением новой функции
- Когда задача сложная и непонятно как подступиться
- Когда меняются бизнес-требования
- Когда принято архитектурное решение
- Когда нашли и решили ошибку

### Когда НЕ нужно обновлять:
- Мелкие правки и баги
- Очевидные изменения
- Когда уже всё понятно и задокументировано

### Формат записей в changelog:
```
## [Дата]: Краткое описание
### Что сделано: ...
### Ключевые решения: ...
### Состояние: ...
```

### Формат записей в errors-and-solutions:
```
### [Дата] Краткое описание проблемы
Проблема: ...
Причина: ...
Решение: ...
Урок: ...
```
